## 1. 요약 (Executive Summary)

현재 관찰된 성능 저하(지시준수 0%, 코드/수학 0%, Elo 하위권)는 단일 원인이 아니라,

1) **학습/운영 파이프라인 정합성 문제**와  
2) **모델 스케일 및 데이터 분포 한계**

가 동시에 작동한 결과로 판단된다.

따라서 개선은 아래 순서가 가장 합리적이다.

1. **평가/서빙 신뢰도 복구** (측정 오류 제거)  
2. **1.2B 응급 복구** (지시준수/코드 0점 구간 탈출)  
3. **Distillation 기반 중간 모델(1.8~2.2B) 확보**  
4. 필요 시 **True 3B 재학습**으로 상한 돌파

## 3. 핵심 진단

## 3.1 코드/파이프라인 측면

1. **Pretrain에서 config train 섹션 반영 불완전**
   - 실제 실행값과 YAML 의도값이 어긋날 가능성.

2. **`--val_data` 전달 대비 pretrain validation 경로 미연결**
   - 런처는 val를 넘기지만 실학습 검증 루프는 비활성에 가까움.

3. **SFT 동적패딩 `attention_mask` 생성 후 모델 미반영**
   - 패딩 구간 영향이 남아 품질 저하 위험.

4. **DDP validation 중복 실행 가능성**
   - 불필요한 시간 낭비 및 스루풋 저하.

5. **런처 스크립트의 `pipefail + grep` 체인 취약**
   - 정상 학습도 false failure로 종료될 리스크.

6. **ORPO 설정 단일 소스 불일치**
   - YAML 존재에도 런처 하드코딩 우세.

7. **데이터 모듈 소스 외부 의존 (`data/` ignore)**
   - 재현성 및 회귀검증 난이도 높음.

## 3.2 평가/결과 측면

1. **v2가 대부분 트랙에서 하한 고정**
   - 양자화(Q4/Q8/F16) 차이가 작고 모두 하위권.

2. **코드/수학 0%**
   - 단순 약점이 아닌 능력 붕괴 신호.

3. **지시준수 0%, 환각탐지 0~20%**
   - 제품화 관점에서 치명적.

4. **멀티턴 대화 1점대**
   - 실사용 대화 유지 실패.

5. **`<unk>` 반복 샘플 관찰**
   - 토크나이저/템플릿/학습 분포 불일치 가능성.

6. **평가 리포트 간 누락/집계 불일치**
   - 일부 문서의 트랙 누락, 평균 계산 불일치 흔적.

7. **비교 클래스 불균형**
   - 평가 모델 실파라미터 1.2B vs 비교군 2.4~4B.

## 4. 전략 원칙

1. **측정 신뢰도 먼저 고정**: 측정이 흔들리면 튜닝이 무의미.  
2. **0점 구간 먼저 탈출**: 코드/수학/지시준수의 바닥 제거가 1순위.  
3. **단기/중기/근본개선 분리**: 응급복구와 재학습 트랙을 혼합하지 않음.  
4. **Gate 기반 의사결정**: 단계별 통과 기준 미달 시 즉시 트랙 전환.

## 5. 개선 대안 후보군

## 시나리오 A: 1.2B 응급복구 트랙 (빠른 회복)

### 목표
- 지시준수 0% -> **35% 이상**
- Track4(코드/수학) 0% -> **10~15% 이상**
- Elo ~1000 -> **1080+**

### 조건 수정
- SFT 데이터에 구조 출력(JSON/표/리스트) 고품질 샘플 증량
- ORPO beta 완화: `0.25 -> 0.05~0.10` 재스윕
- SFT/프리트레인 리플레이 20~30% 혼합 유지
- 서빙 템플릿을 학습 템플릿과 1:1 정렬

### 비용/기간
- 1~2주, 저~중간 GPU 비용

### 성공확률
- **중상 (약 60~70%)**

---

## 시나리오 B: 코드·수학 능력 패치 트랙 (LoRA/Adapter)

### 목표
- Track4 평균 **25%+**
- Track1 하락폭 **2%p 이내**

### 조건 수정
- 코드/SQL/디버깅/수학 검증 가능 데이터셋 집중 미세조정
- 베이스 모델 고정 + LoRA 다중 어댑터 실험
- 추론 시 도메인 어댑터 라우팅

### 비용/기간
- 5~7일, 저비용

### 성공확률
- **중간 (약 50~60%)**

---

## 시나리오 C: Distillation 중간모델 트랙 (1.8~2.2B)

### 목표
- Track2 **4.0+/10**
- Track4 **30%+**
- Elo **1250+**

### 조건 수정

- 대화/추론/코드/수학 균형 distill
- preference 재학습(경량 DPO/ORPO) 결합

### 비용/기간
- 2~4주, 중간 비용

### 성공확률
- **중간 (약 55~60%)**

---

## 시나리오 D: Tokenizer 전환 트랙 (SPM -> BPE)

### 목표
- `<unk>` 반복 및 비정상 토큰 패턴 제거
- 한국어/코드 혼합 도메인 토큰 효율 개선

### 조건 수정
- 새 vocab 설계(예: 80k~128k)
- 전체 데이터 재토크나이즈
- 최소 SFT+정렬 재학습, 가능하면 재프리트레인

### 비용/기간
- 3~6주, 중~고비용

### 성공확률
- **단독 중하 (약 35~45%)**,  
  Distillation/재학습과 결합 시 상승.

## 7. 권장 로드맵 (단일안)

## 단계 0: 평가 신뢰도 복구 (3~5일)
- Track 누락/집계 불일치 제거
- 동일 조건 3회 반복 측정(분산 확인)
- 서빙 템플릿/토크나이저 무결성 점검

### Gate 0
- 트랙 커버리지 100%
- 주요 지표 변동폭 ±3%p 이내
- `<unk>` 비정상 반복 재현률 0에 수렴

---

## 단계 1: A+B 병행 (1~2주)
- 1.2B 응급복구 + 코드/수학 패치
- ORPO beta 및 데이터 혼합 재탐색

### Gate 1
- Track2 >= 2.5/10
- Track4 >= 15%
- 지시준수 >= 0.35
- Elo >= 1080

미달 시 단계 2로 즉시 전환.

---

## 단계 2: C Distillation (2~4주)
- 1.8~2.2B 타깃 중간모델 구축

### Gate 2
- Track2 >= 4.0
- Track4 >= 30%
- Elo >= 1250

달성 시 v2.5 후보. 미달 시 단계 3.

---

## 단계 3: E (+D 결합) True 3B (6~12주)
- 근본 개선 트랙 수행

### Gate 3 (출시 후보)
- Track1 >= 70%
- Track4 >= 45%
- 지시준수 >= 0.65
- 환각탐지 >= 0.55
- Elo >= 1450

## 8. 즉시 적용 가능한 코드/설정 수정 체크리스트

1. `pretrain.py`에 YAML `train` 반영 (lr/save/eval/use_amp/max_grad_norm)
2. pretrain validation 경로 실제 연결
3. SFT/ORPO 런처 파이프라인 종료코드 처리 안정화
4. ORPO 런처를 `--config` 중심으로 단일화
5. DDP validation 중복 제거
6. `attention_mask`를 model attention에 실제 반영
7. 체크포인트 keep 정책 configurable화
8. FP8 strict/auto 모드 명시

## 9. 최종 제언

현재 상태에서는 “더 오래 학습”보다도,

1) **평가/서빙 정합성 복구**와  
2) **지시준수·코드능력의 바닥 탈출**

이 ROI가 가장 높다.

즉, 권장 우선순위는 **A+B -> C -> (필요 시) E**이며,  
D(토크나이저 전환)는 단독 추진보다 C/E와 결합하는 편이 안전하다.
