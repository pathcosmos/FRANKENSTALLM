# 한국어 중심 코딩 LLM 설계 초안 (속도 우선 + 실무형 AI/ML/Python)

> 작성일: 2026-03-13  
> 목적: “빠르고, 한국어를 잘 이해하며, AI/ML + Python 코딩 + Linux/Shell 기본기를 갖춘 코딩 LLM”의 초기 설계안  
> 중점: 데이터셋 조합 전략 + 앙상블 기준

---

## 0) 한 줄 목표

한국어로 요청받아 Python/AI/ML 코드를 빠르게 생성하고, macOS/Linux 환경에서 shell/프로세스/네트워크 문제까지 실무적으로 해결하는 경량 코딩 LLM 구축.

---

## 1) 요구사항 정리

요청사항을 모델 목표로 정리하면 다음 6개입니다.

1. 응답 속도 우선(낮은 지연, 높은 tok/s)
2. 한국어 이해/표현 품질 우선
3. Python + AI/ML 실무 코딩 능력
4. shell/리눅스/네트워크/프로세스 기본기
5. 사용자 의도 파악 + 출력 형식화
6. 위험 명령어/파괴적 작업에 대한 안전성

---

## 2) 모델 제품 포지셔닝

### 2.1 타깃 시나리오

- PyTorch/Transformers 학습 코드 디버깅
- 데이터 전처리/학습/평가 스크립트 작성
- conda/venv/환경변수 문제 해결
- Linux 서버 운영 명령 가이드(로그, 포트, 프로세스)
- 네트워크 점검(접속, 방화벽, DNS, 포트 충돌)

### 2.2 비타깃(초기 제외)

- 초고난도 수학 특화 모델
- 법률/의료 전문 상담
- 완전 자율형 장기 에이전트(초기엔 보조형)

---

## 3) 라인업 제안 (속도/정확도 분리)

1. Fast-Coder-KO (주력)
   - 1.5B~3B Dense
   - 한국어+Python+shell 특화
   - 실시간 질의 대부분 처리

2. Deep-Coder-KO (보조)
   - 4B~7B Dense 또는 상위모델 distill
   - 복잡한 디버깅/긴 컨텍스트/검증 재생성 담당

권장 운영 전략: Fast 기본 + Deep 조건부 승격.

---

## 4) 데이터셋 조합 전략 (핵심)

## 4.1 Pretrain 토큰 믹스

권장 총량:
- 최소 60B tokens
- 권장 120B~180B tokens

권장 비율(v1):

| 데이터 축 | 비율 | 목적 |
|---|---:|---|
| 한국어 일반/지식 텍스트 | 30% | 한국어 이해/표현 |
| 한국어 기술 문서/튜토리얼 | 15% | 기술 설명 품질 |
| 코드 코퍼스(Python 중심) | 25% | 코딩 능력 |
| Shell/Linux/네트워크 문서+로그 | 15% | 운영 문제 해결 |
| 영어 기술 문서/코드 보완 | 10% | 최신 기술 커버 |
| 수학/통계/ML 이론 | 5% | AI/ML reasoning 바닥선 |

코드 코퍼스(25%) 내부 비율:

| 코드 하위축 | 비율 |
|---|---:|
| Python(AI/ML/데이터) | 60% |
| Shell/Bash | 15% |
| SQL | 10% |
| JS/TS(툴링) | 10% |
| 기타 | 5% |

---

## 4.2 후보 데이터 소스 (수집 우선순위)

### A. 한국어 일반/기술 텍스트
- C4 Korean, CC100 Korean, HPLT Korean
- 한국어 위키/기술 블로그(라이선스 허용 범위)
- 기존 프로젝트 한국어 코퍼스(이미 수집 자산 재활용)

### B. 코드 데이터
- 퍼미시브 라이선스 OSS 코드(MIT/BSD/Apache 우선)
- Python AI/ML 레포(훈련, 추론, 배포, MLOps)
- README-코드 병렬 쌍
- 정제된 주피터 노트북

### C. Shell/Linux/네트워크
- man/tldr 계열 문서
- bash/zsh 튜토리얼
- systemd/journalctl/ss/ip/iptables/nft 등 운영 문서
- 장애 로그 -> 원인/조치 매핑 데이터

### D. SFT/대화 데이터
- 한국어 instruction/QA
- 코딩 지시 데이터(문제-해결-검증 구조)
- “요구사항 요약 + 코드 + 실행 + 검증” 포맷 샘플

### E. Preference 데이터
- 좋은/나쁜 답변 쌍(지시준수/안전성/정확성)
- shell 위험 명령 회피와 안전한 대안 제시 샘플

---

## 4.3 데이터 포맷 표준

### 코딩 답변 포맷
1) 문제 요약  
2) 코드  
3) 실행 커맨드  
4) 검증 방법  
5) 실패 시 대안

### 운영/쉘 답변 포맷
1) 원인 가설 1~3  
2) 진단 명령  
3) 복구 명령  
4) 롤백/주의사항

### 의도 확인 포맷
- 정보가 부족한 경우 먼저 재질문(버전, OS, 로그, 목표)

---

## 4.4 품질 게이트 (필수)

1. 라이선스 게이트: 배포 가능 라이선스만 허용
2. 중복 제거: exact + near-dedup(minhash/simhash)
3. 오염 제거: 벤치마크 정답 누출 제거
4. 안전 필터: 악성 명령/비밀정보/PII 제거
5. 실행 검증 샘플: 일부 코드는 실제 실행 통과 데이터로 보강

---

## 4.5 학습 단계별 믹스

Stage A: Pretrain  
- 4.1 비율 유지, 장기 토큰 학습

Stage B: SFT  
- 권장 샘플 비율:
  - 코딩 지시 35%
  - shell/운영 25%
  - 한국어 일반 지시 20%
  - AI/ML 심화 20%

Stage C: Preference(ORPO/DPO)  
- 지시 미준수/환각/위험 shell 제안 hard negative 강화

## 5) 앙상블 설계 기준 (핵심)

## 5.1 앙상블 목표

- 대부분 요청은 Fast 모델로 즉시 처리
- 실패 가능 요청만 Deep 모델로 승격
- 비용 증가를 최소화하면서 품질을 안정화

## 5.2 추론 아키텍처

1. Router(의도 분류)
   - 클래스: 일반, Python, AI/ML, shell, 네트워크/프로세스, 고난도
2. Fast 모델 1차 생성
3. 신뢰도 점수 계산(logprob/형식준수/정책체크)
4. 임계치 미달 시 Deep 모델 재생성
5. Reranker로 최종 선택

---

## 5.3 라우팅 기준(권장 임계치)

| 조건 | 기준 예시 | 액션 |
|---|---|---|
| 요청 길이 | > 900 토큰 | Deep 승격 후보 |
| 다단계 요구 | 파일 3개 이상 수정/설계 포함 | Deep 우선 |
| 시스템 위험도 | sudo/rm/chmod/chown/network reset 포함 | 안전 모드 + 검증 강화 |
| 불확실성 | avg logprob 낮음, 자기모순 탐지 | 재생성 또는 Deep 승격 |
| 형식 불일치 | 요청 포맷(JSON/표) 미준수 | 재생성 |

---

## 5.4 앙상블 유형 제안

### 유형 A: 모델 레벨 앙상블 (권장)
- Fast 1개 + Deep 1개
- 라우팅과 재랭킹으로 비용 최적화

### 유형 B: Adapter 앙상블
- Base 1개 + LoRA 다중:
  - LoRA-Python
  - LoRA-AI/ML
  - LoRA-Shell/Linux
  - LoRA-Network
- Router가 adapter 선택 또는 soft-merge

### 유형 C: Self-Consistency 앙상블
- 동일 모델에서 온도 다르게 2~3개 후보 생성
- 룰 기반 + reranker로 최종안 선택
- 코드/명령 정확도가 중요한 경우에만 사용

---

## 5.5 Reranker 기준(점수 함수)

최종 점수 = 0.35 * 정확성 + 0.20 * 지시준수 + 0.20 * 안전성 + 0.15 * 실행가능성 + 0.10 * 한국어 자연성

세부 점검:
- 코드 블록 완결성
- 명령어 문법/OS 적합성
- 파괴적 명령 경고 포함 여부
- 요청한 출력 형식 일치 여부

---

## 6) 한국어 중심 성능 강화를 위한 데이터 조건

1. 한국어 instruction 비중을 SFT에서 최소 60% 유지
2. 영어 원문 기반 코딩 데이터는 한국어 설명을 병렬 생성(합성 가능)
3. 한국어 기술 질문 의도 분류 데이터 별도 구축
4. 한국어 오류 로그/질문 변형(오탈자/구어체) 증강
5. 한국어 honorific/반말/간결체 스타일 제어 데이터 포함

## 8) 학습/추론 하이퍼파라미터 초안

### Pretrain
- context: 8k(초기), 이후 16k 확장
- optimizer: AdamW
- precision: bf16(기본), 가능하면 fp8 실험
- grad accumulation + checkpointing 적용

### SFT
- packed + dynamic padding
- 지시 준수 포맷 손실 가중치 강화
- shell 안전성 관련 샘플 oversampling

### Preference
- ORPO/DPO 중 1개 고정 후 ablation
- 실패 유형별 hard negative 구성

---

## 9) 평가 설계 (필수 벤치)

1. 코딩
   - HumanEval/MBPP 계열 + 사내 한국어 코딩셋
2. AI/ML 실무
   - 모델 학습/디버깅 태스크셋(자체 구축)
3. Shell/Linux
   - 명령 생성 정확도, 위험 명령 안전성 지표
4. 한국어 지시 준수
   - 포맷 준수율, 요구사항 누락률
5. 속도
   - TTFT, tok/s, p95 latency

---

## 10) 출시 기준(Go/No-Go)

아래 5개를 모두 만족하면 베타 배포:

1. 한국어 지시 준수율 >= 90%
2. Python 코드 실행 성공률 >= 65%(사내 테스트셋 기준)
3. shell 위험 명령 안전 처리율 >= 98%
4. p95 지연 <= 목표값(예: 2.5s 첫 토큰)
5. 사용자 평가에서 “실무 도움 됨” >= 4.2/5

## 11) 8주 실행 로드맵(초안)

### 1~2주
- 데이터 라이선스/품질 파이프라인 확정
- Router 라벨 스키마 정의
- 평가셋/금지셋(오염 방지) 고정

### 3~4주
- Pretrain 1차(소규모) + 중간평가
- SFT 데이터 포맷 정규화

### 5~6주
- SFT + Preference 정렬
- Router + Fast/Deep 앙상블 연결

### 7~8주
- 성능 튜닝(속도/품질/안전성)
- 베타 릴리스 기준 점검

## 부록 B) 앙상블 선정 원칙 요약

1. 기본은 Fast 단일 처리
2. 실패 가능성이 높을 때만 승격
3. 승격 기준은 “복잡도 + 불확실성 + 위험도”
4. 최종 출력은 reranker/정책 체크 통과본만 반환
5. 모델 성능보다 운영 일관성(안전/형식/재현성)을 우선
